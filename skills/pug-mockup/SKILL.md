# Pug Mockup

> Generate SPA mockups with Pug — fast prototyping for stakeholder review.
> Self-contained HTML output: no server, no dependencies, open anywhere.

---

## Why Pug for AI Mockups?

| | HTML | Pug |
|---|---|---|
| **Token efficiency** | Verbose — closing tags, quotes everywhere | ~40-50% fewer tokens — indent-based, no closing tags |
| **AI error rate** | Easy to break nesting with missed `</div>` | Indentation = hierarchy — structurally enforced |
| **Component reuse** | Copy-paste | Mixin + include — edit once, apply everywhere |
| **Data-driven** | Manual duplication | `each` loop — 1 template renders N items |
| **Output** | The source file itself | Compiles to 1 self-contained HTML (CSS/JS inlined) |
| **Tooling** | None needed | Only `pug-cli` (~2MB) — no heavy frameworks |
| **Production path** | Is the production code | ~60-70% reusable in React/Next.js (CSS, structure, logic) |

Pug lets AI write more UI in fewer tokens, with fewer bugs, and the output
works offline as a single HTML file. It's the fastest path from idea to
clickable prototype.

**When to use:** 5-30 page mockups, demo flows, stakeholder review, interactive prototypes.
**When not to use:** 100+ pages, real auth/API needed → use a production framework.

## Prerequisites

- **Node.js** (any recent version)
- **pug-cli**: `npm install -g pug-cli` or use `npx pug-cli` without installing
- Check: `pug --version`
- Pug docs: [pugjs.org](https://pugjs.org/api/getting-started.html)

## Input

- Design specs, wireframes, screenshots, or text description of what to build
- Works best with a UI/UX planning skill that provides layout direction first

## Output

- Pug project: `src/` (source) + `dist/index.html` (compiled)
- Self-contained: copy `dist/` folder — runs offline, no server needed

---

## Flow

1. Read input (design specs, wireframes, or description)
2. Copy the `template/` folder as project starter
3. Define CSS design tokens in `src/css/main.css`
4. Write SPA router + logic in `src/js/main.js`
5. Write components (header, sidebar, footer as mixins)
6. Write pages (1 file per page in `src/pages/`)
7. **Generate `CLAUDE.md`** in project root — page list, components, design decisions (see skill `CLAUDE.md` for template)
8. Build: `pug src/index.pug -o dist -P`
9. Copy assets (images, icons) to `dist/assets/`
10. Review → edit → rebuild (see "How to Iterate" below)

## Project Structure

```
project/
├── package.json          ← build/watch scripts
├── CLAUDE.md             ← project map for Claude (pages, components, decisions)
├── src/                  ← SOURCE — you edit files here
│   ├── index.pug         ← entry point, includes everything
│   ├── components/       ← shared UI (mixins, partials)
│   ├── pages/            ← 1 file per page
│   ├── css/main.css      ← design tokens + styles
│   ├── js/main.js        ← SPA router + interactive logic
│   └── assets/           ← images (1 folder per image, see "Images & Assets")
│       └── [name]/
│           ├── IMAGE.jpg   ← source
│           ├── IMAGE.b64   ← cached base64 (auto-generated)
│           └── _meta.json  ← fingerprint (auto-generated)
└── dist/                 ← OUTPUT — generated by build, do not edit
    └── index.html        ← compiled HTML (open this to view!)
```

**`src/` vs `dist/`:** You edit `.pug` files in `src/`. The build command
compiles them into `dist/index.html` — one self-contained HTML file with
all CSS and JS inlined. Share `dist/index.html` or the whole `dist/` folder.
Never edit files in `dist/` directly — they get overwritten on every build.

## Build Commands

```bash
npm run build            # compile once
npm run watch            # auto-rebuild on file changes
# or without package.json:
pug src/index.pug -o dist -P         # build (pretty-print)
pug src/index.pug -o dist -P -w      # watch mode
```

## How to Iterate

The skill first generates the full project (all pages, components, routing).
After that, you refine **one page at a time**:

1. **Open the specific `.pug` file** you want to edit (e.g., `src/pages/dashboard.pug`)
2. **Select the section** you want to change (e.g., the stat cards grid)
3. **Describe the change** — "add a chart placeholder below the stat cards",
   "change the activity feed to a table", "make the header sticky"
4. **Rebuild**: `npm run build`
5. **Refresh browser** to see changes

**Why one file at a time?** Each `.pug` file is small and focused (one page
or one component). Editing the whole project at once wastes context and risks
breaking pages you didn't want to change. Point to the exact file → describe
the change → get a precise edit.

**Editing components vs pages:**
- Change **how something looks everywhere** → edit the component file
  (e.g., `components/sidebar.pug` to add a new nav item to all pages)
- Change **how something looks on one page** → edit the page file
  (e.g., `pages/dashboard.pug` to rearrange the dashboard layout)
- Change **colors, fonts, spacing** → edit `css/main.css` design tokens

**Example iteration prompts:**
```
# Good — specific file + specific change
"In pages/tasks.pug, add a search bar above the table"
"In components/sidebar.pug, add a 'Reports' section with 2 links"
"In css/main.css, change primary color to #059669 (green)"

# Less effective — vague, touches everything
"Make the app look better"
"Redesign the layout"
```

## Images & Assets

When the mockup needs images (screenshots, diagrams, photos), use the asset
embedding system:

### Structure

```
project/
├── src/
│   └── assets/
│       └── [asset-name]/       ← 1 folder per image, descriptive name
│           ├── IMAGE.[ext]     ← source image (fixed name "IMAGE")
│           ├── IMAGE.b64       ← auto-generated base64 (do not edit)
│           └── _meta.json      ← fingerprint cache (do not edit)
```

### In Pug

Reference by folder name — no file extension:

```pug
img(src="assets/hero-banner" alt="Hero banner" loading="lazy")
img(src="assets/team-photo" alt="Team photo")
```

### Build

The post-build script `embed-images.py` handles everything:

```bash
# In package.json build script, append:
# && python path/to/.claude/scripts/shared/embed-images.py .
```

- **First build:** encodes IMAGE → saves IMAGE.b64 next to source
- **Next builds:** reads cached IMAGE.b64 (skips encoding if unchanged)
- **Warning at 10MB**, **error at 20MB** (need to split assets to dist/assets/)

### Adding images from user

Users send images with random names (e.g., `1775305026733.jpg`, `screenshot.png`).
**Claude must rename to `IMAGE.[ext]`** — never keep the original filename.

1. Create folder — tên mô tả nội dung hình (ví dụ: `hero-banner/`, `team-photo/`, `architecture-diagram/`)
2. Copy image vào folder, **rename thành** `IMAGE.[ext]` (ví dụ: `IMAGE.jpg`, `IMAGE.png`)
3. Reference trong Pug bằng tên folder: `src="assets/hero-banner"`
4. Build → script tự sinh IMAGE.b64

## Self-Contained Output (Export)

Pug's `include` statement inlines CSS and JS directly into the HTML:

```pug
style
  include css/main.css      //- CSS inlined into <style>
script
  include js/main.js        //- JS inlined into <script>
```

The compiled `dist/index.html` is already self-contained — no external
`<link>` or `<script src>` references. Share the single file, open offline.

## Converting Pug to Other Frameworks

Pug mockups are not a dead-end — they're a **fast prototype** that transfers
to any production framework:

| From Pug | To | How |
|---|---|---|
| Built HTML | Any framework | Use `dist/index.html` as visual reference |
| Mixin | React/Next.js component | `+sidebar('x')` → `<Sidebar active="x" />` |
| `include` | ES module `import` | Same file organization, different syntax |
| `main.css` tokens | Tailwind config / CSS modules | Copy design token values |
| Lucide SVG inline | `lucide-react` / `lucide-vue` | `<Search size={18} />` |
| JS router logic | React Router / Vue Router | Same page structure, framework API |
| `each` data loops | `.map()` in JSX | Same data, different iteration syntax |

**Vue bonus:** Vue natively supports Pug in templates (`<template lang="pug">`).

**Reuse rate:** ~60-70% of CSS structure, layout patterns, and component
organization transfers directly. Auth, API, form validation, error handling
are written fresh in the production framework.
